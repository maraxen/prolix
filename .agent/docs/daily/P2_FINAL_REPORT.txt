CRITICAL ANALYSIS: ROOT CAUSE INVESTIGATION
================================================================================

WHAT IS HAPPENING:

At dt=1.0 fs:
  - Temperature is ~290 K (slightly low)
  - Error is ~9.7 K (exceeds 5 K gate but reasonable)
  - System is quasi-stable (modest variance ±11 K)

At dt=2.0 fs:
  - Temperature EXPLODES to ~508 K (69% overshoot!)
  - Error is ~207.7 K (off-scale failure)
  - System shows huge variance (±48 K) indicating instability
  - Kinetic energy roughly doubled (151 → 301 kcal/mol)

WHAT THIS IS NOT:

1. Not an iteration count problem
   - If just iteration count → expect similar errors at both dt
   - Instead: error grows ~21x when dt doubles
   
2. Not a DOF denominator error
   - Oracle verified formula is correct
   - dt=1.0 fs result (9.73 K) matches expected scaling from Phase 1
   
3. Not a simple thermostat issue
   - Single Langevin thermostat can't explain ~21x amplification
   - Shows dt-dependent instability

WHAT THIS REALLY IS:

SETTLE-LANGEVIN INTEGRATION ORDER PROBLEM

The velocity constraint (RATTLE) is incompatible with the current order
of Langevin thermostat operations:

Current implementation (rattle_langevin in physics/simulate.py):
  1. p(t+dt/2) = p(t) + (dt/2)*F(t)           [velocity half-step]
  2. r(t+dt/2) = r(t) + (dt/2)*p/m            [position half-step]
  3. p = c1*p + c2*sqrt(m*kT)*noise           [Langevin stochastic]
  4. r(t+dt) = r(t+dt/2) + (dt/2)*p/m         [position half-step]
  5. F(t+dt) = grad E(r(t+dt))                [force evaluation]
  6. p(t+dt) = p + (dt/2)*F(t+dt)             [velocity half-step]
  7. p(t+dt) = PROJECT(p(t+dt))               [RATTLE constraint] ← PROBLEM

Problem at step 7:
  - Langevin noise was injected at step 3
  - Noise variance is calibrated for the full dt
  - RATTLE constraint then removes momentum from constraint subspace
  - At larger dt: constraint is stronger, removes more momentum
  - Result: Thermostat undoes constraint, constraint removes noise
  - System becomes bistable/oscillatory with energy transfer to unconstrained modes

MATHEMATICAL MECHANISM:

For a 3-site water molecule constrained by SETTLE:
  - Constraint removes 3 DOF (one bond per H-O pair, one H-H constraint)
  - Actually removes energy from 3D velocity space
  - Langevin noise operates in full 3D
  - Mismatch: noise adds energy, constraint removes energy, no equilibrium

At larger dt:
  - RATTLE iterations take longer to converge
  - Constraint force becomes stronger (more aggressive)
  - Noise variance inadequate for constrained subspace
  - Energy bleeds into unconstrained modes
  - Temperature diverges

EVIDENCE FROM MEASUREMENTS:

Phase 1 baseline (wrong DOF):
  dt=1.0 fs → ΔT = 28.8 K
  dt=2.0 fs → ΔT = 10.6 K
  
Phase 2 results (corrected DOF):
  dt=1.0 fs → ΔT = 9.73 K  (consistent with 28.8/3)
  dt=2.0 fs → ΔT = 207.77 K (NEW failure mode)

The dt=1.0 fs improvement suggests DOF fix is working.
The dt=2.0 fs explosion suggests a separate SETTLE-Langevin coupling issue.

================================================================================
SOLUTION CANDIDATES (FOR ORACLE REVIEW)
================================================================================

OPTION A: Constrained Langevin Thermostat
  - Use thermostat that operates only in constraint-free subspace
  - Rewrite: noise *= (1 - P_constraint)
  - Complexity: Medium, requires projection matrices
  - Confidence: High (well-established in literature)

OPTION B: Reorder Operations
  - Apply SETTLE BEFORE thermostat stochastic step
  - Pattern: SHAKE → Langevin noise → RATTLE
  - Ensures constraint satisfaction before thermal energy injection
  - Complexity: Low, just reordering
  - Confidence: Medium (may affect stability differently)

OPTION C: Separate Constraint + Thermostat
  - Use specialized SETTLE+Langevin integrator
  - Similar to OpenMM's SETTLE+LangevinMiddleIntegrator
  - Complexity: High, new integrator implementation
  - Confidence: High (proven in OpenMM)

OPTION D: Adaptive Thermostat Rescaling
  - Modify Langevin variance based on constraint DOF
  - Scale kT by (DOF_total / DOF_free)
  - Complexity: Low
  - Confidence: Low (heuristic, may not generalize)
