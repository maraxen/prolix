## Debt 760 — connect `simulate.py`'s existing NL+PME engine

**File correction**: the engine is `src/prolix/simulate.py` (top-level, 1228 lines) — a different, older, 509-line minimization+RATTLE runner lives at `src/prolix/physics/simulate.py`; don't conflate them.

### Findings

- jax_md's `NeighborList` (`Dense` format) is a plain pytree with a **fixed static capacity** baked in at `.allocate()`. `.update()` is a pure, jittable, vmap-safe positions→NeighborList function. `run_simulation`'s own docstring (`simulate.py:238-243`) documents `jax.vmap(run_simulation, ...)` as supported usage — direct prior art for vmapping over replicas exactly as `EnsemblePlan` does.
- `make_energy_fn` (`src/prolix/physics/system.py:29-262`) already has a working NL-consuming branch (`system.py:175-218`, dispatching to `chunked_lj_energy_nl`/`chunked_coulomb_energy_nl`).
- `dispatch_n_steps_inference` (`src/prolix/api/ensemble_dispatch.py:162-226`, `lax.while_loop` at line 218) has a fully generic `(step_i, state)` carry — no changes needed there for a bigger carry.
- `EnsemblePlan._run_stacked_dispatch` (`src/prolix/api/ensemble_plan.py:349-433`) wraps one `@jax.jit` around `vmap(replicas) × dispatch_n_steps_inference`; `_setup_integrator` (`ensemble_plan.py:463-505`) builds `settle_langevin`'s `init_fn`/`apply_fn` from `energy_fn_from_bundle(bundle)`; `step_fn` (`ensemble_plan.py:610-611`) is the carry-extension point.
- `energy_fn_from_bundle`/`single_padded_energy` (`bundle_md.py:446`, `batched_energy.py:497`) has no `neighbor` parameter today — confirms it's unconditionally dense.
- **What does NOT port**: `simulate.py`'s neighbor-list overflow recovery (`simulate.py:1034-1039`) is a host-Python branch *between* two separate compiled `lax.scan` calls (`simulate.py:1015-1046`) — incompatible with a single-compiled `lax.while_loop`, which has no host round-trip mid-loop.
- **Confirmed still-present, independently relevant bug**: `cutoff_distance` is a named parameter (`system.py:32`) that never reaches `**kwargs`, so the internal tapering read at `system.py:125` (`kwargs.get("cutoff_distance", kwargs.get("cutoff", 9.0))`) silently stays at 9.0 regardless of what's passed. `simulate.py`'s own call (`simulate.py:498-512`) is affected — the neighbor list itself is sized correctly, but the energy function's cutoff-tapering value is not.

### Recommended design

1. Allocate the neighbor list once at bundle-construction time (host-side, generous static capacity via `neighbor_fn.allocate()`).
2. Thread the resulting `NeighborList` as an explicit field of the while_loop's carry `state`.
3. Add an optional `neighbor=` kwarg to `single_padded_energy`/`energy_fn_from_bundle` that dispatches to `chunked_lj_energy_nl`/`chunked_coulomb_energy_nl` — mirrors `system.py:175-218` exactly, no new physics.
4. Wrap `_setup_integrator`'s state in `{langevin_state, neighbor, overflow_flag}`; `step_fn` unpacks, calls `apply_fn(..., neighbor=nbrs, ...)`, periodically (every K steps, mirroring `simulate.py:924-936`'s convention) calls `nbrs.update()`, OR-accumulates `did_buffer_overflow` into the flag instead of branching on it, repacks.
5. After `dispatch_n_steps_inference` returns (a host sync point that already exists), check the accumulated flag once on host — assert/raise on overflow, do not attempt a runtime resize.
6. Fix the `cutoff_distance` kwarg-absorption bug simultaneously — it becomes load-bearing once NL cutoff must physically match tapering.

**No changes needed to `dispatch_n_steps_inference` itself.**

### Open design question (decide before implementation starts)

Neighbor-list capacity depends on local atom density — a variable orthogonal to the atom-count bucket that made 1VII/2GB1 bucket-match this session. Two options:

- **(a) Shared conservative capacity + overflow-check** — simpler, preserves the just-confirmed compile-sharing property, costs some wasted compute. **Recommended default.**
- **(b) New bucket axis keyed on required capacity** — more precise, but risks fragmenting the compile-sharing this session just spent real effort confirming.

Default to (a) unless evidence emerges that it's too lossy in practice — don't undercut the confirmed heterogeneous-batching thesis without cause.

**Effort: 3-5 days. Biggest risk: the capacity-vs-bucket decision above.**

### Open design question — RESOLVED (2026-07-17), with a bigger finding attached

A second, dedicated Opus-backed investigation (fresh agent, full tool access, not a fork) was asked to ground this in real measurements rather than qualitative preference — it built actual `jax_md` neighbor lists on the real 1VII/2GB1 solvated bundles at the codebase's standard 9.0 Å cutoff (`neighbor_cutoff`, `physics/spec.py:19`; `nonbonded_cutoff`, `solvation.py:574`; `system.py:32/125`) and measured, rather than guessed, the answer.

**Measured**: 1VII needs neighbor-list capacity K=428 (real/unpadded positions, `capacity_multiplier=1.25`), 2GB1 needs K=475 — only **~11% apart**, driven almost entirely by bulk TIP3P density (mean occupancy 220 vs 225, nearly identical); the spread comes from each protein's single densest local pocket. Kernel cost is confirmed **linear in capacity** (`T(K) = 7.83ms + 0.060ms·K`, R²=0.99, measured on `chunked_lj_energy_nl`/`chunked_coulomb_energy_nl`) — over-provisioning 2× costs 1.64×, 3× costs 2.46×, a real first-order cost, not negligible. If capacity were derived per-bundle from `allocate()` on real positions, 1VII and 2GB1 **would** land on different static shapes and fragment the compile — confirming option (b)'s risk is real, not theoretical, but cheap to defuse (11% spread, easily covered by one shared value).

**Bigger finding, more important than the original question**: padded/ghost atoms (`atom_mask==False`) are zero-filled by `_pad_2d` (`system.py:387-394`) and therefore all sit at the coordinate origin, *inside* the periodic box. For the 1VII bundle (1963 real / 5000 padded atoms, 61% ghost), this means: (1) a naive `neighbor_fn.allocate()` on the padded bundle needs K≈4118 — a ~10× blowup, ~8× extra kernel cost by the measured linear law; (2) far worse, **212 real atoms (210 for 2GB1) lie within the 9.5 Å effective NL radius of the origin under PBC** — at any sane capacity, ghost-atom entries evict real neighbor entries from real atoms' rows, producing **silently wrong energy/forces**, not just wasted compute; (3) `did_buffer_overflow` becomes useless once ghost rows permanently trip it. This is invisible in the current dense O(N²) path (no capacity concept at all) — genuinely new to debt 760, and would have caused silent physics corruption if 760 had been implemented without catching this. **Filed as debt 772 (P1) — a blocking prerequisite for 760, not an optional nice-to-have.**

### Debt 772 — FIXED (2026-07-17)

Added `_pad_positions_with_ghost_lattice` (`system.py`), replacing the zero-fill (`_pad_2d`) used for the bundle's `positions` field specifically. For periodic bundles (`box_size` present and positive), padding rows are placed on a deterministic, non-overlapping uniform sub-lattice filling the box (`m = ceil(n_ghost^(1/3))` grid points per axis) instead of all coinciding at the origin. Non-periodic bundles fall back to the old zero-fill (no periodic wraparound to place ghosts sensibly against, and no NL capacity concern to protect in free space).

**Scope note**: this fixes only the *initial* placement — the measured problem (a fresh bundle's ghost atoms all at the origin). It does **not** fix ongoing drift: ghost atoms have nonzero mass (`masses_padded`'s `constant_values=1.0`) and `settle.py`'s Langevin O-step (`_langevin_step_o`) has zero `atom_mask` references anywhere in the file — confirmed via grep, not assumed — so ghosts currently *do* receive stochastic momentum kicks and will random-walk away from this lattice over a live trajectory. Deliberately **not fixed here**: touching the core Langevin integrator (extensively validated physics code) to pin ghost atoms is real, separate work that can only be meaningfully tested once debt 760 actually wires a live neighbor list into a running dispatch loop — fixing it in isolation now, with no NL consumer yet to test against, would be unverifiable. Left as an explicit, documented gap for debt 760's dispatch-carry implementation to close (most naturally alongside the same carry-wrapper `step_fn` already being designed for periodic neighbor-list updates).

**Verified**: (1) unit-tested the lattice helper in isolation (unique, in-bounds ghost positions); (2) on the real 1VII bundle, ghost positions are no longer all-zero, all 3037 are unique and within the box; (3) rebuilt the actual jax_md neighbor list on the fixed bundle — capacity dropped from the broken ~4118 to **895** (max occupancy 716, no overflow), matching the predicted ~846-1024 range, not the ~4118 blowup; (4) dense energy is **exactly** invariant to ghost position within a single bundle construction (verified by artificially moving ghosts back to the origin post-construction and comparing energy bit-for-bit — identical, confirming `atom_mask` masking makes ghost position physically irrelevant to the dense path, as expected); (5) **caught and fixed a real regression during verification**: the first version hardcoded `float32` output regardless of input dtype, breaking `test_b1_water_pme_parity.py`'s two `jax_enable_x64=True` tests (`lax.add` dtype mismatch) — `_pad_2d`'s actual behavior (via bare `jnp.pad`, not explicit casting) silently preserves the input's dtype for the non-empty case despite its `dtype=` parameter name, and the replacement needed to match that, not just replicate the misleading-looking `dtype=jnp.float32` call-site argument; (6) 37 tests across the same regression suite used for debt 765 pass; (7) 1VII/2GB1 compile-sharing re-confirmed unregressed (`n_groups=1`, R²=1.0).

**Decision (concrete, not qualitative)**:

- **D1 — capacity formula.** Compute ONE static capacity K per shape-class (never per-bundle, never via `allocate()` on real positions) from bucket constants only: `K_class = ceil_to_multiple_of_128(1.5 × ρ_padded × (4/3)π(r_cutoff + dr_threshold)³)`, where `ρ_padded = atom_bucket_size / box_volume`. For the current 5000-atom/32³-box/9.0 Å class this gives **K=1024** (formula → 822, rounded up for 1.5× headroom over the measured worst case 677 and alignment to both the existing 128 direct-space tile and the 256 flash tile). Because it derives only from shape-class constants, K is bit-identical for every bundle in a class — **fragmentation is structurally impossible**, not just unlikely. Lives as a derived, memoized field keyed on shape-class, computed once at `EnsemblePlan` construction (not per-bundle); plumb an explicit `capacity_multiplier` into `make_neighbor_list_fn` (currently hardcoded 1.25, `neighbor_list.py:159`) or bypass it and pass K directly. `neighbor_fn.allocate()` is used only as a one-time host-side *validator*, never to set the shipped static K.
- **D2 — ghost placement (debt 772, blocking, do first).** Relocate ghost atoms onto a uniform sub-lattice filling the box instead of leaving them at the origin; ensure the integrator holds them fixed (forces already zero via padded sigma/charge=0, but Langevin OU noise on ghost rows must also be masked or they diffuse and re-cluster over a trajectory). With ghosts spread, the measured overhead drops from ~10× to a bounded **~1.75×** (K=846, max_occ=677) — the unavoidable cost of a periodic NL carrying 61%-ghost padding, not a bug. Non-blocking follow-up filed as debt 773: an intermediate `ATOM_BUCKETS` size (~2048, still keeping 1VII/2GB1 co-resident) would reclaim most of that 1.75×.
- **D3 — overflow safety net + ship-gate.** With ghosts spread, no row should overflow at K=1024; keep the already-designed OR-accumulate + host-check pattern, but on a trip **reallocate at 1.5×K and re-run the block** (not assert-and-fail). Before merging 760: run the reference set (1VII, 2GB1, **plus at least one larger/denser solvated case** — two proteins is too thin to fix a global constant on) for N steps and assert (i) `did_buffer_overflow` stays False, (ii) observed max real-row occupancy ≤ 0.66×K (≥1.5× headroom), (iii) energy parity against the current dense O(N²) path (the ground-truth oracle, since it has no capacity to get wrong).

## Phase 6 step 3 — BLOCKED (2026-07-17): a real, pre-existing bug in the NL kernels themselves

Implementing the `neighbor=` kwarg on `single_padded_energy`/`energy_fn_from_bundle` (mirroring `system.py`'s own NL branch, no new physics) surfaced a serious, previously-undiscovered correctness bug — not in the new wiring, but in the shared `chunked_lj_energy_nl`/`chunked_coulomb_energy_nl` kernels (`optimization.py`) themselves, filed as **debt 790 (P1, blocking)**.

**Symptom**: on the real 1VII solvated-protein bundle, `single_padded_energy(..., neighbor=nbrs)` gives `+30262 kcal/mol` vs. the correct dense-path reference of `-2544 kcal/mol` — wrong sign, >10x magnitude error.

**Bisection** (full detail in debt 790): initially suspected debt 772's ghost-atom spreading, but ruled out — ghost atoms have charge=sigma=epsilon=0, so ghost-involving pairs already contribute exactly zero regardless of masking (a defensive ghost-masking redirect was still added, confirmed harmless-but-not-load-bearing here). Systematically isolated via progressively targeted tests: small synthetic cases (N=4 to N=2000, sparse/dense/periodic/multi-tile) all pass exactly; real molecular data at N=100-1963 with **trivial** exclusions all pass; the SAME N=1963 real-data test with the **real**, debt-765-populated per-atom-row exclusion table fails (`-3862` vs. the correct `-106131`). This pins the bug to the kernels' internal exclusion-matching/tile-accumulation logic (`optimization.py`'s `f_tile` closures + `tiling.py`'s `tile_reduction_nl`'s nested `lax.scan`) — confirmed NOT the exclusion data itself (`_build_dense_exclusion_scales` on the same per-atom-row data reproduces the already-validated dense exclusion matrix exactly, 25M/25M entries). A per-atom spot-check (one real atom, 12 exclusion partners) showed every individual pair-scale value computed correctly — the bug likely lives in cross-tile aggregation, not any single atom's local computation, but the exact root cause was not found before stopping.

**Decision**: stopped here rather than continue an open-ended debugging session, or worse, wire a silently-broken energy kernel into `EnsemblePlan`'s production dispatch. The `neighbor=` wiring itself is committed (correct design, natural landing point once fixed) but is prominently documented as non-functional and MUST NOT be used until debt 790 lands. **Steps 4-7 of Phase 6 (carry restructuring, ghost pinning, the OpenMM parity test, the compile-sharing regression check) are all blocked on debt 790** — building any of them on top of a known-broken energy kernel would be wasted, risky work.

## Debt 761 — connect `single_padded_force`/`flash_explicit_forces`

### Correction to the debt's original premise

`flash_explicit_forces` is **not** closed-form analytical — it computes forces via `eqx.filter_grad` (still autodiff, just static-field-safe, the same fix class as this session's tracer-leak bug). True closed-form analytical gradients (`src/prolix/physics/analytical_forces.py`) exist only in `single_padded_force`'s `use_flash=False` **legacy** path, which needs the same dense (N,N) exclusion matrices as today's `single_padded_energy` — no memory win there, and it's not what "flash" defaults to. The PME custom_vjp benefit (`make_spme_energy_fn`) is **already present** in today's `single_padded_energy`+`jax.grad` path — it is not unique to flash.

What flash *actually* changes: `chunked_explicit_nonbonded_energy` (`src/prolix/physics/flash_explicit.py:51-83`) tiles the direct-space LJ+Coulomb pass into 256×256 blocks under `jax.checkpoint` — a genuine memory-locality optimization, but **an unmeasured hypothesis, not a proven win**.

### Findings

- **Wiring mechanics are small and low-risk**: `settle_langevin` (`src/prolix/physics/settle.py:1097`) already auto-detects energy-vs-force via `jax.eval_shape`-probing (`canonicalize_force`/`make_force_fn_like_canonicalize`) — the same pattern already used for this session's `pme_alpha`/`box_size` vmap-safe fallbacks. **No changes needed in `settle.py`.** Just add `force_fn_from_bundle(bundle)` mirroring `energy_fn_from_bundle` (`bundle_md.py:446-475`), and swap the call site in `_setup_integrator` (`ensemble_plan.py:463`).
- **Term coverage**: no missing nonbonded term found for the PME+SETTLE explicit-solvent case (LJ, damped direct-space Coulomb, PME reciprocal, PME exclusion correction, LJ dispersion tail all present in `flash_explicit.py:459-491`).
- **New risk found, filed as debt 763, independent of 761's fate**: `chunked_explicit_nonbonded_energy` computes `n_tiles = N // T` (integer division, `flash_explicit.py:83`, `T=256`) — if a bundle's padded atom count is not an exact multiple of 256, remainder atoms are silently dropped from all nonbonded interactions. Same bug class as the already-documented `optimization.py::inner_tile_size` tiling bug in project `CLAUDE.md`.
- **Test coverage is essentially nonexistent**: `tests/test_analytical_forces.py`'s only reference to `single_padded_force` is an importability check (`assert callable(single_padded_force)`); no numeric test exercises `use_flash=True`/`explicit_solvent=True`, let alone a bundle-derived `PhysicsSystem`. A parity test (flash forces vs. the already-validated `single_padded_energy`+`jax.grad`, on the existing 1VII/2GB1 bundles) is required, not optional, before trusting this in production — budget for finding at least one more real bug here given this session's 4-for-4 track record of verify-before-trust catching real issues.

### Recommendation: measure before committing

Run a cheap, bathos-tracked benchmark (`scripts/experiments/profile_b1_flash_vs_autodiff_forces.py`, campaign `32d6574e`) comparing `single_padded_energy`+`jax.grad` vs. `single_padded_force(use_flash=True)` wall-clock on the existing 1VII/2GB1 bundles *before* deciding whether to wire this into `EnsemblePlan` at all. If the benchmark shows no material win, close 761 as measured-not-worth-pursuing and keep only debt 763 (tile-size bug) open independently.

**Effort: wiring itself ~1-2 hours; full scope (parity test + tile-size check/fix) 1-2 days if the benchmark warrants it.**

### Post-scoping runtime finding (2026-07-17): debt 761 is BLOCKED, not just unmeasured

Running the benchmark (L2 CPU, then a bathos-tracked run `cb191665-a6c4-4552-95d8-892f1956f7b4`, outcome `fail`) surfaced a decisive result *before* any speed number was even reachable: `single_padded_force(sys, disp_fn, implicit_solvent=False, explicit_solvent=True, use_flash=True)` **crashes** on the real 1VII bundle-derived `PhysicsSystem` — both under `jax.jit` (`ConcretizationTypeError` from `flash_explicit.py:439`'s `float(mean_l)`) and unjitted/eager (`IndexError: tuple index out of range` at `flash_explicit.py:114`, `jax.lax.dynamic_slice(sys.excl_scales_vdw, (start_idx, 0), (T, sys.excl_scales_vdw.shape[1]))` — `.shape[1]` on a 1D array).

**Root cause (filed as debt 765, P1 — supersedes this section's earlier "measure benefit" framing as the primary blocker)**: `chunked_explicit_nonbonded_energy`/`_compute_tile_inner` (`flash_explicit.py:89-123`) expects `sys.excl_indices`/`excl_scales_vdw`/`excl_scales_elec` in a **per-atom-row** layout — shape `(N, max_excl_per_atom)`, sliced per-tile via `dynamic_slice`. `physics_system_from_bundle` (`bundle_md.py:440-442`) instead populates these fields in the **pair-list** layout — `excl_indices` shape `(n_pairs, 2)`, `excl_scales_vdw`/`excl_scales_elec` shape `(n_pairs,)` — the same layout `single_padded_energy`'s dense path (`_build_dense_exclusion_scales`) and the PME exclusion correction (`_pme_exclusion_correction_from_pairs`) both correctly consume. This is a genuine, structural incompatibility between two different exclusion-array conventions in the codebase, not a small bug — `flash_explicit_forces` was evidently built and tested against a different, older `PhysicsSystem` construction path that populated the per-atom-row layout, and was never reconciled with the bundle-derived path.

**Consequence**: debt 761 cannot proceed to wiring, or even to a completed speed measurement, until debt 765 is fixed (either build a per-atom-row exclusion array from the bundle's pair-list at construction time — a new `max_excl_per_atom` bucket-relevant constant — or rewrite `chunked_explicit_nonbonded_energy`'s exclusion handling to consume the pair-list format directly). This raises 761's real effort beyond the "1-2 days" estimated during read-only scoping — that estimate did not (and, being code-reading rather than execution, could not) catch this runtime-only shape mismatch. Baseline context captured regardless: on CPU, 1VII (1963 real / 5000 padded atoms), `single_padded_energy` forward-only = 285ms, +`jax.grad` (current EnsemblePlan path) = 1259ms.

**Updated recommendation**: do not invest further time in 761 (wiring or re-benchmarking) until debt 765 is resolved as its own piece of work. Debt 763 (tile-size-256-divisibility) remains a separate, independent, still-unverified risk on top of this — both must be addressed before flash forces can be trusted on bundle-derived systems.

### The dependency the original scoping missed

### A second gap: no PME-over-neighbors term exists yet, anywhere

`single_padded_energy_nl`'s Coulomb branch is explicitly dense, vacuum-only (`pme_alpha=0`, no PME reciprocal) — confirmed in code, not previously known. Debt 760, to actually deliver the PME+SETTLE explicit-solvent case this session's heterogeneous-batching work is centered on, needs a **new damped-direct-space Coulomb-over-neighbors term** (only the per-atom-row `chunked_coulomb_energy_nl` exists) composed with the reusable PME reciprocal correction, **plus a PME-under-neighbor-list parity test** against the already-confirmed dense baseline. This is real new physics wiring, not "no new physics" as originally scoped.

### Other corrections

- No jax_md escape hatch exists for in-loop capacity resize (confirms the original "host-check, don't resize" call) — but `make_neighbor_list_fn` **hardcodes `capacity_multiplier=1.25`** (`neighbor_list.py:159,168`), so "generous static capacity" requires explicitly plumbing a higher multiplier, not assuming the default is generous.

## Debt 802 — RESOLVED (2026-07-18): NL now supports stacked/vmapped multi-bundle dispatch

Following the scope-gap discovery above, debt 802 was implemented in the same session (user: "Please continue").

**Core engineering obstacle**: `neighbor_fn.allocate()` is host-only (it inspects concrete positions to size the cell list) and cannot run on a traced position array inside a vmapped function body. Confirmed empirically that two independent `.allocate()` calls on *different* position arrays (same box/cutoff/N) produce genuinely different capacities (168 vs 176 in one test) *and* non-matching pytree metadata (fresh Python closures per call) — meaning naive per-bundle allocation followed by `jax.tree.map(jnp.stack)` fails outright, not just on shape mismatch.

**Fix, verified empirically before wiring in**: allocate one shared "seed" `NeighborList` host-side, once, before any vmapping (with `extra_capacity` bumped to reach the shape-class's `compute_nl_capacity()` target). Derive each bundle's own `NeighborList` via `seed.update(bundle_positions)` rather than an independent `.allocate()` call — `.update()` is defined as `self.update_fn(position, self, **kwargs)`, i.e. it reuses `self`'s (the seed's) static metadata (capacity, cell-list closures) applied to new positions. Every bundle's result therefore shares *identical* pytree structure with the seed and with each other — directly stackable via `jax.tree.map(jnp.stack)`, and safe to slice per-replica under `jax.vmap` (confirmed: vmapped `.update()` on a stacked NeighborList matches individual per-replica `.update()` calls exactly).

**Implementation** (`ensemble_plan.py`, commit `3950f4c`):
- `dispatch_n_mols` (`ensemble_dispatch.py`) gains an optional `extra_stacked` parameter, backward compatible (all existing 2-arg callers unaffected).
- `_run_single_inference` gains `initial_neighbor`: when provided (the stacked path), skips internal allocation/retry (impossible under vmap tracing) and surfaces overflow via `observable_values["_nl_did_overflow"]` for the caller to check.
- `_run_stacked_dispatch` gains `use_neighbor_list`/`nl_update_every`, builds the seed + stacks per-bundle NeighborLists via two new helpers (`_build_stacked_neighbor_seed`, `_reallocate_stacked_neighbor_seed`), and retries the whole stack once on aggregate overflow (host-checked after the vmapped call returns, matching the single-bundle retry convention).
- Threaded through `_run_grouped`/`_run_inference`/`run()`; the earlier single-bundle-only restriction is relaxed. The rare defensive per-bundle fallback loop inside `_run_grouped` (shouldn't normally trigger — same shape_spec should always stack) now raises clearly if NL is requested there, rather than silently degrading compile-sharing.

**A second custom_vjp/tracer bug found and fixed during verification** (same class as the `neighbor_idx` fix from Phase 6 steps 4-5): `excl_indices` was also marked `nondiff_argnums` on `chunked_lj_energy_nl`/`chunked_coulomb_energy_nl` — harmless when bundle data is concrete (the single-bundle case), but under the *outer* vmap here, a per-replica bundle's `excl_indices` becomes a tracer too. Moved to a differentiable-position argument (zero-gradient placeholder, since gradient w.r.t. an integer index array is always zero) in both kernels, mirroring the existing `neighbor_idx` fix exactly.

**Verified**: two real, distinct (jittered-copy) periodic 1VII bundles run a genuine vmapped NL inference dispatch with finite output over multiple steps; overflow-then-reallocate recovers cleanly under a deliberately undersized capacity; the dense (`use_neighbor_list=False`) stacked path is bit-unaffected. Bisection (isolate bonded terms via the shared helper functions both paths import from `batched_energy.py` → isolate PME reciprocal/background/exclusion-correction individually, each reasonable in magnitude → isolate the LJ dispersion-tail correction) found the exact cause: all three call sites of `explicit_corrections.lj_dispersion_tail_energy` in `flash_explicit.py` (EFA path, EFA_LEBEDEV path, default PME path) pass `n_tail` — a **scalar real-atom count** — as the function's 5th positional argument, which is actually named `atom_mask` and expected to be an `(N,)` per-atom boolean/float array. Passing a bare count broadcasts it as a scalar multiplier into `sum_c6`/`sum_c12` (each already squared), inflating the term by roughly `n_atoms²` (~1963² ≈ 3.85M× for 1VII) — a catastrophic but silent blowup (finite, not NaN/Inf, so nothing crashed).

**This exact bug, with the exact same failure mechanism, was already found and fixed in `batched_energy.py` (lines 510-528)** in an earlier session — that fix even has a comment documenting the `n_atoms²`-scaling mechanism precisely. It was simply never propagated to `flash_explicit.py`'s three copy-pasted call sites. Fixed by passing `sys.atom_mask` at all three sites, mirroring the already-fixed pattern exactly.

**Consequence**: every flash-forces wall-clock number measured in this document and in `scripts/experiments/profile_b1_flash_vs_autodiff_forces.py` **before this fix** was measuring a computation returning wildly wrong energy/force values — the benchmark never checked correctness, only that it ran without crashing and how long it took. The "3.5-3.6x speedup" numbers recorded earlier in this session are real wall-clock measurements of a broken computation.

### Debt 805 — ROOT-CAUSED AND FIXED (2026-07-19): missing periodic minimum-image convention

Root cause of the residual ~5.5% discrepancy above: `chunked_explicit_nonbonded_energy`/`_chunked_lj_only_energy` compute `dr = pos[:, None, :] - r_j[None, :, :]` as a **raw Cartesian subtraction with no periodic wrapping at all**. The dense reference (`_lj_energy_masked`/`_coulomb_energy_masked`) instead computes `dr` via `space.map_product(displacement_fn)(r, r)`, where `displacement_fn` comes from `pbc.create_periodic_space(box_size)` (`jax_md.space.periodic`) — i.e. genuine minimum-image distance. `pbc.py` already has a standalone `minimum_image_distance` helper (`dr - round(dr/box)*box`) used elsewhere in the codebase; this exact convention was simply never applied inside flash's tiled kernel.

**Confirmed empirically before fixing** (not just by code inspection): on the real 1VII bundle (32 Å periodic box), **54.06% of real-atom pairs** have a raw Cartesian separation that differs from their true periodic minimum image in at least one dimension. Recomputing the dense direct-space energy by hand with (a) raw Cartesian `dr` (matching flash's old behavior) and (b) minimum-image-wrapped `dr`, then comparing to the actual dense reference: raw gives -1825.59, wrapped gives -1959.03, dense (`displacement_fn`) gives -1959.03 — wrapped matches dense to 4 decimal places, raw reproduces almost exactly the previously-measured gap. This is not a rounding-level bug: for a periodic explicit-solvent system, it silently gives the **wrong distance** (and therefore wrong energy AND wrong force direction) for the majority of pairs, systematically increasing in severity for any atom near a box face — this would have been far more consequential for `EnsemblePlan` forces than the ~5.5% total-energy number suggests, since individual per-atom force errors near boundaries are not bounded by the aggregate energy discrepancy.

**Fix**: both tiling functions now wrap `dr` with `dr - sys.box_size * jnp.round(dr / sys.box_size)` immediately after computing it, gated on `sys.box_size is not None` (a static field, so this is a Python-level branch resolved at trace time, matching the same `is not None` convention already used elsewhere in this file for periodicity checks — not a `jax.lax.cond`).

**Verified**: re-ran the full real-1VII-bundle comparison after the fix — `flash_explicit_total_energy` vs `single_padded_energy`: **-2520.10 vs -2520.06 (rel diff 1.3e-5)** — float32-precision agreement, down from 5.46%. LJ-only and Coulomb-only isolated comparisons (undamped, `pme_alpha=0`) both agree to <0.001 out of ~1000-4000 (previously ~3.8%/~7.1% off). The synthetic (non-periodic, `box_size=None`) parity tests validating debt 763's fix are unaffected (still match to ~1e-4, as expected — no wrapping applies when `box_size is None`).

## Debt 760 — cross-protein NL compile-sharing: CONFIRMED via single-protein control jobs (2026-07-19)

Following the debt-803 cluster-sync remediation (git-based workaround, above), the actual `--mode combined --use-neighbor-list` cross-protein run (job `18330082`, `engaging`) completed successfully: `n_groups_from_partition=1`, `compile_fixed_s=12.036s`.
