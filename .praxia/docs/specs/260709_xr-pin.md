# XR-PIN

## Goal
Single reproducible **PyPI** xtrax pin across main, worktrees, CI, and Engaging; CI fails if import version is below the floor.

## Decision (2026-07-09) — resolves T1

| Field | Value |
|-------|--------|
| Install source | **PyPI only** (no `file://` path-dep) |
| Constraint | `xtrax>=0.4.0a5,<0.5` |
| Floor string | `0.4.0a5` |
| Compare rule | **lower bound** via `packaging.version`: installed `>=` floor and `< 0.5` |
| Rationale | Latest published xtrax is `0.4.0a5` (matches local tree). One pin everywhere; if xtrax needs unpublished work, revisit with an explicit exception. |

## Scope
- `pyproject.toml` / `uv.lock`: replace path-dep with PyPI constraint above
- Align xtrax-030-cutover worktree to the same constraint (drop `>=0.3,<0.4`)
- CI / local check: PEP 440 compare against floor `0.4.0a5` and upper `<0.5`
- Document in epic TBD T1 as resolved

## Non-goals
Changing planner behavior; Sprint A physics; publishing a new xtrax release (0.4.0a5 already on PyPI).

## Acceptance Criteria
1. Given a clean checkout of main **without** a sibling `~/projects/xtrax`, when `uv sync` completes, then the resolved xtrax version satisfies `>=0.4.0a5,<0.5` (source = PyPI, not path).
2. Given the pin check runs (`uv run python scripts/ci/check_xtrax_pin.py` or equivalent CI step), when `xtrax.__version__` parses below `0.4.0a5` or `>=0.5`, then the process exits non-zero; when version is `0.4.0a5` or a later 0.4.0aN / 0.4.x below 0.5, then it exits zero.
3. Given the xtrax-030-cutover worktree `pyproject.toml`, when compared to main, then both declare the same `xtrax>=0.4.0a5,<0.5` constraint (no documented exception).
4. Given `hasattr(xtrax, "__version__")` is false, when the pin check runs, then it fails closed (non-zero).
5. Given a path/editable xtrax install (`direct_url.json` file: or dir_info) or a `uv.lock` xtrax `source` that is not `{ registry = ... }`, when the pin check runs, then it exits non-zero even if the version string is `0.4.0a5`.
6. Given Engaging/cluster checkouts sync this repo's `pyproject.toml`/`uv.lock`, when they install deps, then they receive the same PyPI pin (no separate Engaging constraint).

## Rollback
Revert `pyproject.toml` / `uv.lock` + pin-check script/CI step.
